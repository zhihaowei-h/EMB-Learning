# 守护进程 (Daemon) 核心编程指南

## 📖 什么是守护进程？
守护进程（Daemon）是 Linux/Unix 系统中一种运行在后台的特殊进程。它独立于控制终端，并且周期性地执行某种任务或等待处理某些发生的事件。
常见的守护进程有系统日志进程 `syslogd`、Web 服务器 `nginx`、定时任务 `crond` 等。它们通常在系统引导装入时启动，在系统关闭时终止。

守护进程的核心特征：
1. **后台运行**：脱离终端控制（没有关联的 TTY）。
2. **生命周期长**：通常 7×24 小时不间断运行。
3. **自成一派**：它是新会话（Session）的组长，也是新进程组（Process Group）的组长。

## 🚀 代码演进脉络 (File by File)
本项目的 5 个源码文件完整展示了“**如何将一个普通程序一步步改造成工业级的守护进程**”的全过程。

### 阶段一：业务原型的诞生
👉 **`01mydaemon.c`**
* **功能**：这是一个基础的业务程序。它每隔 1 秒获取当前系统时间，并将其格式化写入到 `/tmp/out` 文件中。
* **缺陷**：它是一个普通的前台程序。如果你在终端运行它，终端会被霸占；如果你关闭终端（或按下 `Ctrl+C`），程序就会被系统强制杀掉。它完全依赖于当前的终端环境。

### 阶段二：手动蜕变 —— 标准守护进程的 5 步曲
👉 **`02mydaemon.c`**
* **演进**：引入了自定义的 `mydaemon()` 函数，手动实现了将普通进程转化为守护进程的标准套路。
* **核心步骤**：
  1. `fork()` 并终止父进程：让子进程在后台运行，并保证子进程不是进程组长（为 `setsid` 做准备）。
  2. `setsid()`：创建新会话，脱离原终端的控制，成为新的会话组长。
  3. `umask(0)`：清除文件权限掩码，防止创建文件时权限被意外限制。
  4. `chdir("/")`：将工作目录切换到根目录，防止占用可卸载的文件系统（如挂载的 U 盘）。
  5. **重定向标准 I/O**：将 `0` (stdin)、`1` (stdout)、`2` (stderr) 全部重定向到 Linux 的黑洞 `/dev/null`。因为守护进程脱离了终端，往屏幕打印没有任何意义，甚至可能引发 `SIGTTOU` 等信号导致程序崩溃。

### 阶段三：化繁为简 —— 拥抱系统 API
👉 **`03mydaemon.c`**
* **演进**：放弃手动造轮子，使用 Linux 提供的标准库函数 `daemon(0, 0)` 替换了自己写的 `mydaemon()`。
* **意义**：在实际工程中，除非有特殊的定制需求，否则直接使用 `unistd.h` 提供的 `daemon()` 函数是最安全、最简洁的做法。代码中通过 `#if 0 ... #else ... #endif` 清晰地对比了两种实现方式。

### 阶段四：企业级特性 —— 单实例保护 (Single Instance)
👉 **`04single_instance_mydaemon.c`**
* **演进**：加入了 `process_already_running()` 函数，引入了**文件互斥锁（File Locking）**机制。
* **痛点解决**：守护进程通常只允许在系统中运行一个实例（比如你不能同时跑两个负责写同一个日志文件的程序，否则会数据错乱）。
* **技术点**：利用 `open(LOCKFILE, O_CREAT)` 打开位于 `/var/run/`（或 `/tmp/`）的 `.pid` 文件，并使用 `flock(fd, LOCK_EX | LOCK_NB)` 加上**非阻塞的排他锁**。如果加锁失败，说明已经有一个实例在运行，当前程序立即退出。加锁成功后，将自己的 `PID` 写入文件方便后续管理。

### 阶段五：系统级融合 —— 规范化日志输出 (Syslog)
👉 **`05syslog.c`**
* **演进**：引入了 `<syslog.h>`，彻底抛弃了 `perror()` 和 `fprintf(stderr, ...)`。
* **痛点解决**：从第二阶段开始，我们的标准输出和错误输出就已经被丢进 `/dev/null`（黑洞）了。如果程序内部发生错误（比如加锁失败、创建会话失败），开发者根本看不到报错信息，变成了“瞎子”。
* **技术点**：使用 `openlog()` 连接系统日志服务，遇到错误时使用 `syslog(LOG_ERR, ...)` 发送日志。这样，程序的运行异常就会被规范地记录到 `/var/log/syslog`（或 `/var/log/messages`）中，方便运维人员排查问题。

## 🛠️ 编译与运行测试指南

**1. 编译命令**
```bash
gcc 05syslog.c -o mydaemon_app
```

**2. 运行测试**
由于阶段 4 和 5 需要在 `/var/run/` 下创建锁文件，通常需要管理员权限：
```bash
sudo ./mydaemon_app
```

**3. 现象观测**
* 运行后终端立即返回，程序进入后台。
* 使用 `ps -ef | grep mydaemon_app` 可以看到该进程，且 `TTY` 列显示为 `?`（表示脱离终端）。
* 查看业务输出：`tail -f /tmp/out`，会看到时间在不断更新。
* 尝试启动第二个实例：`sudo ./mydaemon_app`。
* 查看报错日志：`cat /var/log/syslog | grep daemon`，你会看到 `syslog()` 发出的报错信息（如提示锁已被占用）。
