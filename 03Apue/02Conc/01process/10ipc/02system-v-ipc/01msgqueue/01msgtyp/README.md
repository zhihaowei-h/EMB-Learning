# System V 消息队列 `msgtyp` 参数用法演示

本目录包含了 4 个独立的 C 语言源文件，旨在演示 Linux System V 消息队列中 `msgrcv()` 函数的核心参数 —— **`msgtyp` (消息类型)** 的四种不同读取策略。
## 📊 `msgtyp` 参数详解表

| msgtyp | 读取方式 | 说明 |
| :--- | :--- | :--- |
| **msgtyp == 0** | 按 FIFO(先进先出)的方式读取 | 无视消息的类型，直接读取队列最头部 (最早放入) 的第一条消息。 |
| **msgtyp > 0** | 精准匹配 (最常用) | 读取队列中第一条类型等于 `msgtyp` 的消息。 |
| **msgtyp < 0** | 优先级 / 范围读取 | 当传入负数时，操作系统会取该负数的绝对值。然后，在队列中寻找类型值小于或等于这个绝对值的消息。如果有多个满足条件的消息，它会优先读取类型值最小的那一条。 |

---

所有的示例程序都在初始阶段向队列中按相同顺序发送了 3 条消息：
1. **Type = 3** ("Type 3 Data") —— *最早入队*
2. **Type = 1** ("Type 1 Data")
3. **Type = 2** ("Type 2 Data") —— *最后入队*

## 📁 示例文件详解

### 1. 绝对的 FIFO (先进先出)
* **文件:** `01_msg_fifo.c`
* **策略:** `msgtyp = 0`
* **行为:** 系统无视消息的类型，直接读取队列最头部（最早放入）的第一条消息。它让消息队列退化成了普通的管道。
* **预期结果:** 读取到 **Type=3** 的消息。

### 2. 精准匹配提取 (最常用)
* **文件:** `02_msg_exact.c`
* **策略:** `msgtyp > 0` (示例中 `msgtyp = 2`)
* **行为:** 精确狙击，直接读取队列中第一条类型完全等于 `msgtyp` 的消息，无视排在前面的其他类型消息。常用于多进程通信中的“频道分离”。
* **预期结果:** 读取到 **Type=2** 的消息。

### 3. 优先级范围提取
* **文件:** `03_msg_priority.c`
* **策略:** `msgtyp < 0` (示例中 `msgtyp = -3`)
* **行为:** 系统寻找队列中类型值 **小于或等于** `|msgtyp|` (绝对值) 的消息，并优先读取其中**类型值最小**（即优先级最高）的那一条消息。
* **预期结果:** 满足 `<= 3` 的有 Type 3、1、2。其中最小的是 1，因此读取到 **Type=1** 的消息。

### 4. 排除特定类型读取 (Linux 特性)
* **文件:** `04_msg_except.c`
* **策略:** `msgtyp > 0` 且 flag 包含 `MSG_EXCEPT` (示例中 `msgtyp = 3`)
* **行为:** 读取队列中排在最前面的、且**类型不等于** `msgtyp` 的消息。
* **预期结果:** 第一条 Type 3 被跳过，读取到第二条进入的 **Type=1** 的消息。

## ⚠️ 核心踩坑笔记

1. **消息结构体的硬性要求**：
   无论你传递什么数据，自定义的消息结构体**第一个成员必须是 `long` 类型的类型标志 (`mtype`)**。`msgrcv` 底层是严格依靠强转指针来读取这个 `long` 字段进行路由判断的。

2. **宏定义冲突 (`04_msg_except.c`)**：
   在使用 Linux 专属的 `MSG_EXCEPT` 标志时，必须在文件顶部定义 `#define _GNU_SOURCE`。这会导致 `<sys/msg.h>` 暴露出系统内置的 `struct msgbuf`。为了防止“重定义”报错，我们必须将自定义的结构体改名（例如改为 `struct my_msgbuf`）。

3. **资源清理**：
   System V IPC 资源（包括消息队列、共享内存、信号量）的生命周期是随内核的，而不是随进程的。如果在程序结束时没有调用 `msgctl(msgid, IPC_RMID, NULL)`，该队列会一直残留在系统中，直到重启或使用 `ipcrm` 命令手动删除。
