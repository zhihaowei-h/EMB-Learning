## 共享内存实验：System V 与 mmap 对比

### 1. 文件功能说明

* **`01shm.c` (传统派)**：
    使用 **System V IPC** 机制。它在内核中申请一块名为“共享内存段”的对象。进程需要通过 `shmget` 得到 ID，再通过 `shmat` 把这块内核对象“挂载”到自己的虚拟地址空间。
* **`02mmap.c` (现代派)**：
    使用 **虚拟内存映射** 机制。通过 `MAP_ANONYMOUS` 标志，直接在物理内存中划出一块区域，并将其映射到进程的虚拟地址空间，不依赖任何内核 IPC 对象。

---

### 2. 核心关系与差异



| 维度 | `01shm.c` (System V) | `02mmap.c` (mmap 匿名映射) |
| :--- | :--- | :--- |
| **共享源** | **内核 IPC 对象** (存在于内核空间) | **物理内存页** (直接分配的物理内存) |
| **同步方式** | 父子进程分别显式挂载 (`shmat`) | 子进程通过 `fork` **继承**父进程的映射 |
| **清理责任** | **随内核**：程序关了内存还在，必须 `shmctl` 销毁 | **随进程**：进程结束或 `munmap` 后自动回收 |
| **关系本质** | 两个进程去连接同一个“后台 ID” | 两个进程共用同一套“物理地址翻译表” |

---

### 3. 实验结论

* **相同点**：两者最终都指向了**同一块物理内存**，实现了“零拷贝”通信，是速度最快的进程间通信方式。
* **不同点**：`mmap` 匿名映射更简洁、更安全（不会造成资源永久泄漏），但它**必须**依靠 `fork` 的血缘关系来传递内存指针；而 System V 机制虽然繁琐且需要手动清理，但它可以通过 `key` 让完全无关的进程也实现接头。

**简而言之：** `02mmap` 是利用“血缘继承”实现共享，而 `01shm` 是利用“公共编号”实现共享。
